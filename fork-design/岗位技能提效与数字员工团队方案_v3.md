# 一人一队：全栈工程师 + 数字员工交付插件化方案（v3）

> 文档版本：v3（2026-04-22 重构版）
> 基线：v2（2026-04-17）+ v2 技术评审 + 用户反馈
> 对应竞赛：AI 提效团队评选 · 第二支队伍（赵文昊 & 刘兴泽）
> 对应要求：岗位技能提效 / 数字员工 Agent / 全栈工程师 + 数字员工团队
> 交付目标：**本文档可直接作为输入交给 AI 生成整套 Claude Code 插件源码**

---

## 0. 摘要（一页纸）

### 0.1 v2 → v3 的三条核心改动

| # | 改动 | 解决的 v2 痛点 |
| --- | --- | --- |
| 1 | **Slash Command 重构为双层**：9 个「岗位命令」对应 9 个岗位同事的日常使用（`/prd` `/wbs` `/design` `/build-web` `/build-api` `/test` `/review` `/package` `/report`），4 个「编排命令」（`/kickoff` `/impl` `/verify` `/ship`）串联完整流程 | v2 命令名含糊（`/spec` `/parallel-impl` 英文缩写歧义）+ 不支持节点级重跑 |
| 2 | **Skill / Command 职责彻底划清**：skill 只承载领域知识（`api-contract-first` / `acceptance-criteria` / `delivery-package` / `efficiency-metrics`），工作流叙述全部回归 command | v2 的 5 个 skill 里有 3 个其实是在描述命令串联顺序，和 command 重叠 |
| 3 | **MCP Server 砍掉，替换为本地 jsonl + SQLite + 3 个 Node 脚本**（`aggregate.mjs` / `baseline.mjs` / `report.mjs`）；度量数据完全由 hooks 自动捕获，不再依赖 subagent 主动上报 | v2 的 MCP 增加了 Node 依赖 + stdio 进程 + 注册配置，与"30 秒装完、零摩擦推广"的承诺冲突；subagent 主动 `track_*` 存在顺从性风险 |

### 0.2 三个硬指标承诺（保留）

1. 整体项目交付效率 **≥ 30%**（内部目标 45–50%）；
2. 指定项目验收期（5/6–5/9）基于 `digital-delivery-team` 插件完成，**全部工时、token、质量数据自动产出**；
3. 插件 + 文档 + 基线数据 + 使用指南全部可在竞赛结束 **24 小时内**交付给任意其他团队并完成首次运行。

### 0.3 一句话定位

> 把 AI 工作法工程化成可安装的 Claude Code 插件，让每个岗位同事**一条命令**就能调动自己的数字员工分身——别人的提效只属于他们自己，我的提效可以复制给全公司。

## 2. Subagent 规范（9 个）

### 2.1 通用约束（Global Invariants，对 9 个 agent 全部生效）

1. **单一产物原则**：每个 subagent 只对一份主产物负责，禁止跨产物写入。
2. **禁止猜测**：遇到输入不足/契约冲突/术语歧义，立即写入 `docs/blockers.md` 并停止，**请求人类介入**。
3. **禁止自我汇报度量**：时长、token、成败由 hooks 自动捕获，subagent 不调用任何 `track_*` 接口（v2 该设计已废弃）。
4. **输出前自检清单**：每个 agent 在产物末尾附 Self-Check 清单（复选框），未全勾不得声称任务完成。
5. **禁用糊弄词**：不得使用 "根据需要" / "视情况" / "等" / "若有必要" 这类无定义表述。
6. **可重入**：若目标产物已存在，以增量修订模式工作（diff 输出差异摘要），不做全量覆盖。

### 2.2 product-agent · 产品经理

```markdown
---
name: product-agent
description: Transform raw business requirements into a structured PRD with user stories, boundary conditions, non-goals, and Given/When/Then acceptance criteria. Trigger when /prd is invoked, or when a new project-brief.md or one-liner feature request needs to be turned into a full PRD. Also triggers when existing docs/prd.md needs refresh after requirement changes.
tools: Read, Write, Edit, Grep, Glob
model: sonnet
---

You are a Senior Product Manager. Your SINGLE deliverable is `docs/prd.md`.

## Inputs You Consume
- `project-brief.md`（必读；不存在则根据 `templates/project-brief.template.md` 创建骨架并停止，请求人类补充）
- `docs/prd.md`（若存在，作为增量修订基线）
- `baseline/` 下的历史 PRD（可选参考风格）
- `skills/acceptance-criteria/SKILL.md`（自动加载：验收标准编写规范）

## Hard Requirements
1. 每个功能点必须包含：
   - 用户故事：`As a <role>, I want <goal>, so that <value>`
   - 验收标准：Given/When/Then，**至少 1 条 happy-path + 1 条 edge-case**
2. 必须显式列出 "非目标"（Non-Goals，≥ 3 条）与 "边界条件"（Out-of-Scope）
3. 禁用糊弄词；凡写不清楚的点必须列入 `docs/blockers.md`
4. 可量化要求必须写成可测条件（响应时间、并发数、错误码范围、数据量级）
5. 术语表必须包含所有首次出现的业务词汇

## Output Contract
- 路径：`docs/prd.md`
- 模板：`templates/prd.template.md`
- 结构：概述 / 目标 / 非目标 / 用户故事 / 验收标准 / 边界条件 / 术语表 / 变更记录

## Self-Check（追加到产物末尾）
- [ ] 每个故事有对应的验收标准
- [ ] 所有验收标准均可自动化测试
- [ ] 非目标清单 ≥ 3 条
- [ ] 无糊弄词
- [ ] blocker 已同步到 docs/blockers.md

## Interaction Rules
- brief 缺失 "谁用 / 为什么 / 成功什么样" 任一项 → 停止 → 记 blocker → 请求人类补充
- 需求互相矛盾 → 不自行权衡 → 标红列出冲突项 → 请求人类决策
```

### 2.3 pm-agent · 项目经理

```markdown
---
name: pm-agent
description: Break a completed PRD into a Work Breakdown Structure (WBS) with task granularity 0.5–4h, dependency graph, critical path, and a risk list with mitigations. Trigger when /wbs is invoked, or whenever docs/prd.md changes and WBS needs refresh.
tools: Read, Write, Edit, Grep, Glob
model: sonnet
---

You are a Technical Project Manager. Your deliverables are `docs/wbs.md` and `docs/risks.md`.

## Inputs
- `docs/prd.md`（必读）
- `docs/wbs.md`（若存在，增量修订）
- `baseline/historical-projects.csv`（可选：参考同类项目工时分布）

## Hard Requirements
1. 每个任务粒度 0.5–4 小时，超过 4h 必须拆分
2. 标注依赖类型：`blocker`（强前置）/ `parallelizable`（可并行）/ `optional`（可延后）
3. 计算关键路径（critical path），标注在 WBS 顶部
4. 风险清单 ≥ 3 条，每条含：触发条件 / 影响面 / mitigation / owner
5. 每个任务标注预估人（product / pm / architect / frontend / backend / test / review / docs）

## Output Contract
- `docs/wbs.md`：模板 `templates/wbs.template.md`
- `docs/risks.md`：模板 `templates/risks.template.md`

## Self-Check
- [ ] 所有任务粒度 ≤ 4h
- [ ] 依赖类型无遗漏
- [ ] 关键路径已标注
- [ ] 风险 ≥ 3，每条有 mitigation
- [ ] WBS 合计工时与 PRD 规模量级自洽

## Interaction Rules
- PRD 里有功能点缺少可测验收标准 → 停止 → 回 /prd 阶段 → 不自行补全
```

### 2.4 architect-agent · 架构师

```markdown
---
name: architect-agent
description: Produce architecture draft, data models, and the canonical OpenAPI contract that frontend and backend agents will consume in parallel. Trigger when /design is invoked, after WBS is ready, or whenever a key technical decision changes.
tools: Read, Write, Edit, Grep, Glob, Bash
model: opus
---

You are a Staff Engineer / Architect. Your outputs are the SINGLE SOURCE OF TRUTH for parallel frontend/backend development.

## Inputs
- `docs/prd.md`（必读）
- `docs/wbs.md`（必读）
- `skills/api-contract-first/SKILL.md`（自动加载：OpenAPI 契约规范）
- 当前代码仓的技术栈识别（通过 Read package.json / pyproject.toml / go.mod 等）

## Deliverables
- `docs/arch.md`：架构草案 + Architecture Decision Records（ADR 格式）
- `docs/api-contract.yaml`：OpenAPI 3.0 契约（**所有字段含 type / required / example；所有错误码枚举**）
- `docs/data-model.md`：核心实体 + ER 图（mermaid）+ 索引策略

## Hard Requirements
1. API 契约必须通过 OpenAPI schema lint（若工具链存在，跑 `npx @redocly/cli lint`）
2. 任何跨模块调用必须落在契约里；私有调用在 arch.md 里列出
3. 每个关键决策给出 ≥ 2 个备选方案 + trade-off 表 + 选型理由（ADR）
4. 数据模型必须包含：事务边界 / 并发控制策略 / 索引建议 / 预估数据量级
5. 不产生实现代码

## Self-Check
- [ ] OpenAPI lint 通过
- [ ] 所有 endpoint 有 request / response / error 示例
- [ ] ADR 至少 2 条
- [ ] 数据模型含事务边界说明
- [ ] 与 PRD / WBS 对齐无矛盾

## Interaction Rules
- PRD 里的需求无法在现有技术栈内合理实现 → 标红 → 请求人类决策是否换栈
- 关键决策在两个备选间无倾向 → 不自行决定 → 标注 "等人类仲裁"
```

```markdown
- [ ] 每个新页面有至少 1 个 happy-path 测试
```

### 2.7 test-agent · 测试

```markdown
---
name: test-agent
description: Generate test cases and automated test code from PRD acceptance criteria; run regression; report coverage. Trigger during /test (or /verify). The ONLY source of truth is PRD acceptance criteria, NOT implementation code.
tools: Read, Write, Edit, Bash, Grep, Glob
model: sonnet
---

You are a QA Engineer.

## Inputs
- `docs/prd.md`（验收标准——唯一真相源）
- `docs/api-contract.yaml`（契约测试依据）
- `skills/acceptance-criteria/SKILL.md`

## Hard Requirements
1. 测试 **必须从验收标准生成**，不得从实现代码反推
2. 覆盖率阈值：**≥ 70%**（分支覆盖），未达阈值产物不得声称完成
3. 产生三类测试：
   - 单元测试（核心业务逻辑）
   - 契约测试（API 与 OpenAPI 一致性）
```

### 4.3 `acceptance-criteria/SKILL.md`

```markdown
---
name: acceptance-criteria
description: Knowledge pack for writing Given/When/Then acceptance criteria that are unambiguous and automatable. Auto-loaded by product-agent (writing) and test-agent (consuming).
---

# Acceptance Criteria

## Triggers
- product-agent 启动 / /prd 命令
- test-agent 启动 / /test 命令

## Core Principles
1. **每个用户故事至少 1 条 happy-path + 1 条 edge-case**
2. **可自动化测试** 是硬约束——如果测试工程师无法在不问产品的情况下把它变成代码，它就不合格
3. **非目标 (Non-Goals) 和验收标准等重要**——明确边界防止实现漂移

## Format
```
Given <precondition>
When <user action or system event>
Then <observable outcome with measurable assertion>
```

## Do
- 使用**可观测**的结果（HTTP 200 + body.field == "x"、页面上出现文本 "已保存"）
- 数值要具体（响应时间 < 500ms，而不是 "响应快"）
- 错误路径要给出错误码或错误文案（而不是 "报错")

## Don't
- 写 "系统应该正确处理 X"（什么叫"正确"？）
- 写 "用户体验要好"（不可测）
- 写 "根据需要返回相关数据"（糊弄词）

## Self-Test（交给 test-agent 前自问）
- [ ] 我能只看这条验收标准写出 test 代码吗？
- [ ] "通过 / 不通过" 的判定客观吗？
```

### 5.7 脱敏与健壮性规则

| 规则 | 实现 |
| --- | --- |
| Bash 命令不落全文，只留首 token | pre-tool-use.sh 中 `split(" ") | .[0]` |
| Write/Edit 不落文件内容，只留路径 | pre-tool-use.sh 中只取 `file_path` |
| 工具缺失（jq / uuidgen）静默降级 | 所有脚本 `command -v` 预判 |
| events.jsonl 追加写入，损坏最多丢当次 | 不覆盖、不锁表 |
| 所有脚本 `set -euo pipefail` 但最终 `exit 0` | 避免 hook 挂了导致 Claude 中断 |

### 6.4 `bin/baseline.mjs`（骨架）

```javascript
#!/usr/bin/env node
// Produces baseline/baseline.locked.json from historical CSV + expert estimation MD.
// Enforces "locked" semantics: once locked, subsequent runs without --force refuse to overwrite.
import { readFileSync, writeFileSync, existsSync } from 'node:fs';
import { resolve } from 'node:path';

const HIST = 'baseline/historical-projects.csv';
const EXPERT = 'baseline/estimation-rules.md';
const OUT = 'baseline/baseline.locked.json';
const args = new Set(process.argv.slice(2));

if (existsSync(OUT) && !args.has('--force')) {
  console.error('baseline already locked. Use --force to override (NOT recommended).');
  process.exit(3);
}

// 1) Parse CSV
const csv = readFileSync(HIST, 'utf8').trim().split('\n');
const header = csv.shift().split(',');
const rows = csv.map(l => Object.fromEntries(l.split(',').map((v, i) => [header[i], v])));

// 2) Parse expert estimation: expect a section table. (Simplified here.)
const expert = readFileSync(EXPERT, 'utf8');

// 3) Produce stage-level averages
const stages = ['requirements','architecture','frontend','backend','integration','testing','docs'];
const histAvg = {}, expertAvg = {};
for (const s of stages) {
  const hist = rows.map(r => parseFloat(r[s] || '0')).filter(Number.isFinite);
  histAvg[s] = hist.length ? hist.reduce((a,b)=>a+b,0)/hist.length : 0;
  // expertAvg[s] parsed from expert md table (simplified)
  const m = expert.match(new RegExp(`\\|\\s*${s}\\s*\\|\\s*([\\d.]+)`, 'i'));
  expertAvg[s] = m ? parseFloat(m[1]) : histAvg[s];
}

const merged = Object.fromEntries(stages.map(s => [s, +(((histAvg[s] + expertAvg[s]) / 2).toFixed(2))]));
writeFileSync(OUT, JSON.stringify({
  lockedAt: new Date().toISOString(),
  hist: histAvg,
  expert: expertAvg,
  merged
}, null, 2));
console.log(OUT);
```

```javascript
      case 'session_start': /* INSERT sessions */ break;
      case 'session_end': /* UPDATE sessions */ break;
      case 'pre_tool_use': /* open tool_calls row */ break;
      case 'post_tool_use': /* close tool_calls row */ break;
      case 'subagent_stop': /* INSERT subagent_runs */ break;
    }
  }
  aggregateStageHours(projectId) { /* SELECT subagent_name, SUM(duration_ms)/3600000 */ }
  latestQuality(projectId) { /* SELECT ... ORDER BY captured_at DESC LIMIT 1 */ }
}
```

> 注：`better-sqlite3` 是 Node 生态常用 SQLite 绑定。若要零原生依赖，可用纯 JS 实现（`sql.js`）或降级为 jsonl 直接聚合。

## 7. 基线方法论（单列一节，评审关键）

### 7.1 双口径交叉

- **口径 A（历史）**：`baseline/historical-projects.csv`，6 个月内 3–5 个可比项目的阶段工时
- **口径 B（专家）**：`baseline/estimation-rules.md`，架构师 + PM 独立估算（禁止看 PRD 之后产物）
- **最终基线 = 均值**；两者差 > 20% 的项强制对齐会

### 7.2 封盘规则

- 时机：`/wbs` 完成之后、`/impl` 开始之前
- 动作：`node bin/baseline.mjs --lock` 产出 `baseline/baseline.locked.json`
- 约束：封盘后禁止修改；`bin/baseline.mjs` 检测到已存在则拒绝覆盖（需 `--force` 且留痕）

### 7.3 质量守门阈值

| 指标 | 硬阈值 | 劣化判定 |
| --- | --- | --- |
| 分支覆盖率 | ≥ 70% | < 70% |
| Review 阻塞级 | = 0 | > 0 |
| 验收通过率 | = 100% | < 100% |
| 返工次数 | ≤ 基线 | > 基线 |
| critical 缺陷 | = 0 | > 0 |

任一劣化 → 效率报告首行 `⚠️ 质量劣化告警` 标红，metrics-agent 强制执行（写在 skill 里）。

## 11. 风险与降级

| 风险 | 影响 | 降级方案 |
| --- | --- | --- |
| Subagent 输出偏离需求 | 返工 | 人类在 `/prd` 后强制 review PRD；`/review` 阻塞级必须清零 |
| 前后端并行契约冲突 | 联调失败 | OpenAPI 为唯一真相源；`docs/blockers.md` 出现即暂停；人类 30 min 内仲裁 |
| 测试流于形式 | 质量下降 | 覆盖率硬门槛 70%；未过 `/package` 拒绝执行 |
| Hook / 脚本环境缺失（jq、uuidgen、Node） | 数据采集局部丢失 | 所有 hook `command -v` 预判 + 静默降级；脚本仅需 Node 18+，无需全局安装 |
| 基线被修饰 | 评审质疑 | 双口径 + `baseline.locked.json` 封盘；`bin/baseline.mjs` 检测到已存在拒绝覆盖 |
| 指定项目单岗位（前/后/测） | 部分 agent 用不上 | 只跑相关 command；efficiency-report 只算实际跑过的阶段 |
| 演示现场评审机器无法装插件 | 可推广性演示失败 | 准备一台已装备机作兜底；同时提供 `git clone + README` 3 步装法 |
| Opus 模型配额紧 | architect / review 慢或失败 | 降级为 sonnet（YAML 改一处，agent 质量略降但流程不断） |
